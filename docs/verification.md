# Verification Ledger

This ledger prevents implementation presence from being reported as working
functionality. Status values are `pending`, `local`, `contract`, `live` and
`production`. A higher status requires evidence from that actual environment.

For requirement coverage and the distinct statuses `implemented-tested`,
`production-baseline-disabled`, `deferred-provider` and `external-blocked`, see
`docs/build-plan-compliance.md`.

The canonical redacted production transcript for the current application release
is [`docs/release-evidence-c0c1544.md`](release-evidence-c0c1544.md).

## Evidence rules

- CI retains backend JUnit/coverage, frontend coverage/Playwright output and local
  image-scan JSON for 14 days. Checked-in synthetic Darwin/Linux baselines use a
  200-pixel visual-diff ceiling, and CI runs the Linux browser gate. The publish
  workflow attaches provenance/SBOM to registry images and retains each exact
  registry-digest scan report for 30 days.
- Tests use stable IDs that can be mapped back to acceptance claims.
- Screenshots and fixtures contain synthetic/redacted data only.
- Mock evidence never satisfies live-provider or production infrastructure claims.
- A production claim requires the revision and immutable image digests.
- Evidence belongs to the exact tested revision. Editing code after a passing run
  returns affected rows to pending until the relevant gate is rerun.
- A readiness 200 is scoped to the dependencies actually probed by that endpoint.
  The current readiness route queries Postgres and, when the async runtime is
  enabled, requires Redis plus fresh worker/scheduler heartbeats. It reports
  provider selectors but does not call Meta, Bumpa or Hermes.

## Release decision

Application release `c0c15443352ab84fde1d2edfde1ed0692ed842f6` meets the
contained production-release bar: exact-revision main CI passed 13/13 jobs, image
publication and exact-digest scans passed 7/7 jobs, all six immutable indexes were
promoted through the guarded coordinator, schema `0015_bumpa_store_context` is
live, and the runtime, edge, brand/SEO, web-login, Bumpa persistence and Hermes
canaries pass within their recorded scopes.

Exactly five approved mapped collaborators can use the country-aware web form and
shared six-digit provider-free temporary PIN. The apex, admin and research hosts
retain independent role gates and host-only sessions; unmapped and wrong-PIN
attempts fail generically. Acceptance cleanup left zero active challenges and zero
active acceptance sessions. WhatsApp remains explicitly parked and disabled.

This decision does **not** authorize unrestricted provider-backed use. Before
WhatsApp activation, complete the Meta sender/template/delivery gates. Before the
remaining provider limitation is called complete, obtain a successful current
`products.overview` response for the fifth store. Encrypted off-host backup and
isolated restore, external alert delivery, and formal privacy/security/retention
approval remain open. The temporary shared PIN also needs rotation/expiry and eventual
replacement with per-user proof of possession.
